Exemplo de Consulta SQL — Métricas DORA por Squad

BigQuery 
SELECT
  t.ano,
  t.semana_ano,
  s.nome_squad,
  COUNT(*)                                    AS qtd_deploys,
  AVG(f.lead_time_total)                      AS lead_time_medio_horas,
  AVG(f.review_time)                          AS review_time_medio_horas,
  COUNTIF(f.fl_revertido) / COUNT(*) * 100    AS change_failure_rate_pct
FROM refined.fato_deploys f
JOIN refined.dim_tempo      t ON f.sk_tempo       = t.sk_tempo
JOIN refined.dim_squad      s ON f.sk_squad       = s.sk_squad
WHERE f.fl_sucesso = TRUE
  AND t.ano = 2025
GROUP BY 1, 2, 3
ORDER BY 1, 2, 3;
Executado no BigQuery · dataset: refined · Orquestrado via Apache Airflow (DAG diária 06h00 BRT)

Camada RAW

Bronze
Ferramenta: Airbyte open-source
Fontes extraídas
  • GitHub API: pull_requests, commits, deployments, workflow_runs, check_runs
  • Jira API: issues, sprints, changelog de status, story_points
  • Frequência: Incremental a cada 6 horas · Full refresh semanal
Schema evolution automático pelo Airbyte. Dados preservados sem transformação. Destino: BigQuery · tabelas prefixadas com raw_. Retenção: 90 dias.

Camada TRUSTED

Silver
Ferramenta: dbt Core + BigQuery
Operações
  • Deduplicação por chave natural (pr_id, deployment_id)
  • Parse e normalização de timestamps para UTC-3
  • JOIN entre pull_request e deployment_event via SHA do commit
  • Cast de tipos e validação NOT NULL em campos críticos
  • Filtro de bots: WHERE author NOT LIKE '%[bot]%'
Tabelas: stg_github__pull_requests, stg_deployments, stg_jira__issues, stg_engineers. Testes dbt: not_null, unique, accepted_values em todos os campos PK/FK.

Camada REFINED

Gold
Ferramenta: dbt models SQL + BigQuery
Transformações-chave
  • lead_time_total = TIMESTAMP_DIFF(deployed_at, first_commit_at, HOUR)
  • review_time = TIMESTAMP_DIFF(pr_merged_at, pr_opened_at, HOUR)
  • coding_time = TIMESTAMP_DIFF(pr_opened_at, first_commit_at, HOUR)
  • periodo_adocao_ia = CASE WHEN data < '2024-01-15' THEN 'pre_ia' ELSE 'pos_ia' END
  • Surrogate keys via ROW_NUMBER() OVER (ORDER BY ...)
Orquestrador: Apache Airflow · DAG diária 06h00 BRT. Consumo: Looker Studio conectado diretamente ao BigQuery.

Por que ELT, não ETL? — 3 Razões Técnicas

🖥

Processamento MPP

BigQuery processa terabytes em segundos via arquitetura massivamente paralela. O motor de transformação é o próprio warehouse, eliminando servidores ETL dedicados e seu custo de manutenção.

💻

Dados Brutos Preservados

A camada raw_layer permanece imutável. Quando a definição de "deploy com falha" é refinada, basta criar um novo modelo dbt — sem reprocessar o pipeline de extração ou perder histórico.

🔍

Audit Log Completo

Toda transformação é um modelo SQL versionado no Git, com testes automatizados e documentação gerada pelo dbt. Qualquer métrica é rastreável até o dado bruto original.

Análise Estratégica — Diagnóstico para o CTO

Por que o Lead Time cresceu após a adoção de IA?

4 hipóteses diagnósticas identificadas — testáveis via SQL no modelo dimensional

Hipótese 01
📈

PRs maiores e mais complexos

Quando o coding_time diminui, engenheiros tendem a incluir mais mudanças em um único PR. Os dados mostram crescimento em qtd_linhas_adicionadas após jan/2024. PRs acima de 800 linhas têm review_time médio 3,4x maior. Solução: limite de 400 linhas por PR como política de engenharia.

Hipótese 02
🪣

Saturação da capacidade de revisão

O volume de pull requests cresceu ~60% após a adoção de IA, mas qtd_revisores permaneceu estável. A fila de Code Review cresce matematicamente. Diagnóstico via SQL: calcular review_time médio por revisor ao longo do tempo. Solução: distribuir revisão e implementar triagem automatizada (SonarQube).

Hipótese 03
🔥

Aumento no CFR por qualidade insuficiente

Código gerado por IA passa em testes unitários mas falha em integrações complexas. O CFR subiu de 14% para 22% — 8 pontos percentuais. A equipe consome tempo crescente em fl_hotfix=true e fl_revertido=true, capacidade que não aparece no Lead Time de novas features mas que a corrói silenciosamente.

Hipótese 04
🔎

Erosão da confiança no código de IA

Revisores que acumularam experiências com bugs de IA passaram a aprofundar suas revisões — resposta racional ao aumento do CFR. O review_time médio cresceu 89% não por ineficiência, mas por necessidade legítima de maior cautela. A solução não é pressão por velocidade, mas melhoria estrutural da qualidade do código gerado.

Conclusão Executiva

Os dados demonstram que a IA Generativa na TechFlow está operando como amplificador de volume, não de valor. O gargalo não está na escrita de código — está na capacidade de revisar, validar e entregar com qualidade.

A solução de BI proposta entrega ao CTO visibilidade cirúrgica sobre onde cada hora é perdida no ciclo de entrega: o modelo dimensional com fato_deploys e suas quatro dimensões transforma dados dispersos do GitHub e Jira em um único ponto de verdade analítico. O pipeline ELT com Airbyte, dbt e BigQuery garante latência baixa, rastreabilidade total e custo controlado. E o dashboard em três níveis permite ir do diagnóstico estratégico ao drill-down de um PR específico em três cliques.

A recomendação é clara: implementar esta estratégia como fundação analítica da gestão de engenharia da TechFlow — onde os dados de ontem alimentam as decisões de amanhã, e onde a IA é avaliada não por promessas, mas por evidências mensuráveis.
